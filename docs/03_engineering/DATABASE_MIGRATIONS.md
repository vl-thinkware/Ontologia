# Database Migrations

Both databases evolve — and both need disciplined migrations. This document covers Postgres (Drizzle) and Neo4j (bespoke Cypher migrator).

---

## 1. Principles

- **Additive first.** Add columns and constraints before removing.
- **Backwards-compatible between deploys.** Old app + new schema must work for one deploy; then new app + new schema; then the cleanup migration.
- **Never in the middle of a request.** Migrations run pre-deploy in a job; app starts only after the migration succeeds.
- **Versioned and signed off.** Every migration has a file, a number, a description, and an owner.

## 2. Postgres — Drizzle

### 2.1 Location

`packages/db/migrations/*.sql` — generated by `drizzle-kit`. Each file: `0001_add_organizations.sql`, etc.

### 2.2 Authoring a migration

```bash
pnpm db:generate --name add_webhook_subscriptions
```

This scans `packages/db/schema.ts` for changes and produces a SQL file. Review the SQL before committing. Drizzle's generator is good, not magical.

### 2.3 Running

- Local: `pnpm db:migrate`
- CI: smoke runs against an ephemeral Postgres on every PR.
- Deploy: a pre-deploy step runs the migrations and fails the deploy on error.

### 2.4 Online-safe patterns

- **Add a column.** Nullable first; backfill via a job; flip `NOT NULL` in a later migration.
- **Rename a column.** `ADD` new, dual-write in app, backfill, switch reads, drop old in a later migration.
- **Indexes.** `CREATE INDEX CONCURRENTLY` whenever the table is non-trivial.
- **Changing types.** Add a new column with target type, backfill, cut over, drop old.
- **Dropping.** Only after a release that stops reading and writing it — minimum 1 full deploy gap.

### 2.5 RLS considerations

New tenant-scoped tables must:
- Include `org_id UUID NOT NULL`.
- `CREATE INDEX ON <table>(org_id)`.
- `ALTER TABLE … ENABLE ROW LEVEL SECURITY`.
- `CREATE POLICY tenant_isolation …`.

Template:

```sql
ALTER TABLE new_table ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON new_table
  USING (org_id = current_setting('app.current_org_id')::uuid)
  WITH CHECK (org_id = current_setting('app.current_org_id')::uuid);
```

### 2.6 Seed data in migrations

Avoid. If unavoidable, gated by env (never in prod). Customer-data seeding belongs in application code / admin tooling.

## 3. Neo4j — Cypher migrator

### 3.1 Why a custom migrator

- Constraints and indexes are essentially "schema".
- We must run both on the shared Neo4j and on every tenant-dedicated database.
- A lightweight homegrown migrator is less code than adopting a general tool.

### 3.2 Location & format

`packages/graph/migrations/*.cypher` — numbered files.

```
0001_initial_constraints.cypher
0002_fulltext_concept.cypher
0003_relation_type_constraint.cypher
```

Each file is an idempotent, single-transaction script. Example:

```cypher
CREATE CONSTRAINT concept_version_id IF NOT EXISTS
  FOR (c:Concept) REQUIRE c.versionId IS UNIQUE;

CREATE INDEX concept_org_ontology IF NOT EXISTS
  FOR (c:Concept) ON (c.orgId, c.ontologyId);
```

### 3.3 Bookkeeping node

Each Neo4j database keeps a tiny meta node:

```cypher
MERGE (m:_Schema { key:'version' })
  ON CREATE SET m.version = 0
RETURN m.version AS v;
```

The migrator lists files `> v`, runs them in order, then `SET m.version = newMax`.

### 3.4 Running

- Local: `pnpm neo4j:migrate`.
- Deploy:
  - Shared DB migrated once per release.
  - Dedicated DBs migrated in parallel by a job (concurrency capped).
- Newly provisioned dedicated DBs run the full chain at provision time.

### 3.5 Authoring an online-safe change

- Adding a label / property: immediate.
- New constraint: consider existing data first; script a precheck.
- Rebuilding an index: `DROP INDEX` + `CREATE INDEX` may be expensive; prefer `CREATE INDEX <new> IF NOT EXISTS` then app-level swap, then drop the old later.

### 3.6 Data migrations (Neo4j)

When a change requires walking the graph (e.g. add `orgId` to edges that previously lacked it), write a Cypher batch script under `packages/graph/data-migrations/` with resumable semantics:

```cypher
MATCH (c:Concept)
WHERE c.orgId IS NULL
WITH c LIMIT 5000
SET c.orgId = c.tmpOrgId
REMOVE c.tmpOrgId
RETURN count(*) AS migrated;
```

Runner loops until `migrated = 0`.

## 4. Versioning migrations across environments

- A migration is assumed deployed to prod **only** after it ran green on staging.
- Migration number collisions (two PRs pick the same next number) are caught by CI.

## 5. Rollback

- **Postgres.** We write *forward-only* migrations. If a migration is wrong, write a new migration that reverts it. `drizzle-kit` can generate the inverse; review carefully.
- **Neo4j.** Same principle. Constraints and indexes can be dropped; data migrations should be reversible where possible.

## 6. Backup before destructive changes

- For Postgres: a named snapshot via the managed provider API, captured before the deploy of any release that drops columns / tables.
- For Neo4j Aura: on-demand snapshot before schema-altering releases.
- Scripts in `ops/backup/` automate both.

## 7. Audit

Every migration file has a preamble:

```sql
-- Migration: 0042_add_published_versions
-- Owner: @alex
-- Date: 2026-06-15
-- Description: adds published_versions table for stable refs.
-- Online safe: yes (additive)
-- Rollback: 0043_drop_published_versions.sql
```
